## 1. 用户行为分析仪表盘

**What:** 建立用户行为追踪和分析仪表盘，可视化各阶段完成率、放弃步骤、付费转化率、信心引擎触发频率。

**Why:** 上线后没有数据驱动的产品迭代能力。需求文档v1.0定义了4个MVP成功标准（10用户完成阶段1、3用户付费、1用户全流程、NPS>7），但没有数据基础设施来测量它们。

**Pros:**
- 实时看到用户在哪个步骤放弃，针对性改进
- 付费转化率数据指导定价策略调整
- 信心引擎触发频率帮助优化提示模板

**Cons:**
- 需要额外的事件存储（SQLite新增events表或集成Mixpanel）
- 前端需埋点，增加app.js复杂度

**Context:** 当前代码中没有任何行为追踪。app.js的localStorage只存用户状态，不发事件。后端SQLite没有events表。需求文档FR-05提到进度展示，但没有追踪层。上线后这应该是第一优先级，因为MVP成功标准全靠它来验证。

**具体追踪字段:**
- `step_complete`: {user_id, skill, step_index, duration_s, confidence_triggers}
- `phase_complete`: {user_id, skill, total_steps, output_summary}
- `paywall_hit`: {user_id, current_tier, attempted_phase}
- `payment_complete`: {user_id, tier, amount}
- `session_start` / `session_end`: {user_id, skill, step_at_start, step_at_end}

**Depends on:** 无外部依赖。可在任何时间点实现。

## 2. 数据库迁移版本管理 (Alembic)

**What:** 引入Alembic管理SQLite表结构变更，替代当前 migrations.py 中的硬编码CREATE TABLE。

**Why:** 上线后有用户数据后，任何表结构变更（加字段、改索引、加表）都需要安全的迁移路径。当前 migrations.py 只支持首次创建，不支持增量变更。手动ALTER TABLE在有数据的情况下风险极高。

**Pros:**
- 每次表结构变更有版本记录，可回滚
- 自动生成迁移脚本（alembic revision --autogenerate）
- 团队协作时数据库变更可追踪

**Cons:**
- 引入Alembic依赖和学习成本
- 对当前单表单文件的MVP略显重量级

**Context:** 当前 db/migrations.py 包含4个CREATE TABLE语句（users, orders, user_profiles, user_states），无版本管理。上线后典型迁移场景：
- user_states表加 `last_active_at` 字段（追踪活跃度）
- users表加 `source` 字段（区分不同渠道来的用户）
- 新增 `events` 表（行为追踪，见TODOS #1）
- user_profiles表加 `phone` 字段（付费用户联系方式）

**实现路径:**
1. `pip install alembic`
2. `alembic init alembic`
3. 配置 alembic/env.py 指向SQLite
4. 将现有 migrations.py 标记为 initial migration
5. 之后每次改表结构用 `alembic revision --autogenerate`

**Depends on:** 无外部依赖。但建议在TODOS #1（行为追踪）之前实现，因为#1需要新增events表。
**Suggested by:** /plan-eng-review (2026-04-28)

## 3. Stage 0 对话状态机设计

**What:** 设计阶段0发现的对话状态机：状态（COLLECTING → ASSESSING → COMPLETED），转换条件（何时判定已收集到3+知识点），对话历史窗口管理。

**Why:** 外部意见（outside voice）指出这是整个产品风险最高的技术项，但当前设计为零。"动态问答序列"只描述了意图，没有指定系统如何检测完成（3+知识点）、如何管理对话上下文、何时停止提问。

**Pros:**
- 状态机设计完成后，实现有明确的行为规范，不需要猜测
- 对话历史窗口设计直接影响token消耗和响应质量

**Cons:**
- 约1小时的设计工作（ASCII状态图 + 转换规则）

**Context:** 这应该在实现步骤2（阶段0对话逻辑）之前完成。没有这个设计，开发者会在实现过程中猜测完成逻辑。outside voice指出v2.0的固定步骤引擎（current_step_index + 1）完全不适用于v4.0的动态对话模式。

**具体设计内容:**
- 状态定义：COLLECTING（收集中）、ASSESSING（评估是否足够）、COMPLETED（完成）
- 转换条件：COLLECTING→ASSESSING = 已问过3+个不同角度的问题；ASSESSING→COMPLETED = LLM确认识别到3+个有效知识点
- 对话历史窗口：保留最近N条消息（N根据token限制动态计算）
- 失败路径：5轮后仍无法提取 → COMPLETED with partial results + 坦诚反馈

**Depends on:** 无。
**Suggested by:** /plan-eng-review outside voice (2026-05-02)

## 4. 参考账号人工整理

**What:** 为主要行业（建材、餐饮、美业等）整理3-5个参考账号，覆盖抖音、小红书、闲鱼，包含粉丝量、内容类型、变现方式。

**Why:** 自动化搜索对中国社交媒体账号的结果不可靠（百度对抖音索引差，Google不索引小红书）。LLM会基于不可验证的搜索结果产生自信但错误的账号推荐，损害用户信任。

**Pros:**
- 高质量、验证过的参考账号，用户可以直接学习
- 按行业整理，针对性强

**Cons:**
- 需要人工调研，每个行业约2-3小时
- 需要持续维护（账号可能停更）

**Context:** 已从MVP中移除（outside voice review + 用户决定）。应在MVP上线后，有真实用户行业分布数据时再做——数据告诉你哪些行业需要优先整理。

**Depends on:** MVP用户反馈数据（行业分布）
**Suggested by:** /plan-eng-review outside voice (2026-05-02)

## 5. LLM客户端容错（熔断器 + 备用方案）

**What:** 给DeepSeek/EvoLink客户端添加熔断器模式。EvoLink连续失败3次后，切换到直接调用DeepSeek API或备用模型（通义千问）。记录失败日志用于监控。

**Why:** EvoLink是整个产品的单点故障。当前客户端零容错——无API失败重试、无备用方案、仅120秒超时。Pydantic重试只处理输出格式错误，不处理API不可用。

**Pros:**
- API不可用时产品不会完全瘫痪
- 日志帮助监控EvoLink稳定性

**Cons:**
- 增加客户端复杂度
- 备用模型可能需要不同的prompt调优

**Context:** outside voice指出这是关键缺陷。MVP阶段（3-10用户）简单的重试（3次指数退避）足够。完整的熔断器可以等。但备用机制（直接DeepSeek API）应该现在设计好，方便后续添加。

**实现路径:**
1. 在LLMClient中添加重试逻辑（httpx已支持，配置retry=3）
2. 配置备用base_url（直接DeepSeek API endpoint）
3. 添加简单的熔断计数器（连续失败次数）
4. 日志记录每次失败的详细信息

**Depends on:** 无。
**Suggested by:** /plan-eng-review outside voice (2026-05-02)
