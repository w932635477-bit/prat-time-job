# 启点 (Starting Point) — 产品需求文档

生成时间：2026-05-02
版本：v3.1.1
状态：APPROVED
前置版本：v1.0 (AI驱动的职业转型教练), v2.0 (经验变现导航器), v3.0 (首单陪跑系统)

## 1. 产品概述

### 1.1 产品定位

启点是一个**经验变现渠道建设系统**，帮助中国40+失业中年人将自己的行业经验/渠道/人脉/资源包装成产品，然后一步步建立销售渠道（抖音、小红书、视频号）。

**不承诺收入结果。** 产品承诺的是过程 — 帮用户走完"发现→包装→建渠道"的完整路径，分阶段推进，每阶段有明确标准。

### 1.2 目标用户

- 主要用户：40-55岁，有10年以上行业经验，因行业结构性衰退失业的中年人
- 典型画像：有可变现的隐性知识（价格、渠道、避坑经验），技术能力有限
- 筛选条件：能清晰表达经验，愿意每天投入30分钟，接受分阶段成长目标

**不是目标用户：**
- 认知有限、无法清晰表达经验的纯体力劳动者
- 完全丧失信心不愿行动的人
- 不愿投入时间、只想"被动等结果"的人（"你先帮我做"型）

### 1.3 核心价值主张

**"帮你的经验建一条从发现到渠道的完整路径。"**

### 1.4 产品形态

微信H5网页应用。核心交互：**对话式AI指导**。

### 1.5 验证案例

**案例1：刘为（正面）**
- 44岁，建材行业老兵，对建材价格和平替产品底价十分熟悉
- 通过对话功能发现自己的经验可以卖钱
- 创始人手动帮做20条抖音视频，总播放量约2500，有客户咨询
- 赚到第一笔咨询费500元
- **但**：视频制作pipeline对他来说太复杂，无法独立完成

**案例2：婚礼策划两姑娘（反面）**
- 在婚礼策划公司做短视频投流，管理约1000万年投流费用
- 得知自己的知识就是产品时非常兴奋
- 持续不到一个月就放弃：脚本需要创始人写，自己拍的视频效果欠佳
- **关键洞察**：启动成本太高+没有持续外部推动

## 2. 六阶段成长路径

### 2.1 阶段总览

| 阶段 | 名称 | 做什么 | 阶段标准（升级条件） |
|------|------|--------|---------------------|
| 0 | 发现 | 识别可变现的经验/资源/人脉 | 能说出3个以上可变现知识点 |
| 1 | 包装 | 把经验包装成产品（话术、定价、服务形式） | 输出完整的产品描述+定价方案 |
| 2 | 冷启动 | 建账号、发首批内容 | 单条播放>500，有自然增长 |
| 3 | 破千 | 稳定输出，优化内容 | 单条播放>1000，出现评论/咨询 |
| 4 | 互动 | 回复咨询、建立信任 | 出现私信/加微信 |
| 5 | 成交 | 完成首次交易 | 第一笔收入（不承诺） |

### 2.2 与旧版本阶段映射

| v3.1 阶段 | v1.0 阶段 | 变化说明 |
|-----------|-----------|----------|
| 0 发现 | 阶段0评估 + 阶段1发现金矿 | 合并评估和发现，聚焦"识别可变现知识" |
| 1 包装 | 阶段2产品包装 | 基本不变 |
| 2 冷启动 | 阶段3获客（部分） | **核心变化**：从"逐日打卡计划"改为"建账号+行动指南+阶段标准" |
| 3 破千 | 无 | **新增**：v1.0没有持续运营阶段 |
| 4 互动 | 阶段4首单（部分） | 从"完成首单工具包"改为"咨询互动训练" |
| 5 成交 | 阶段4+5 | 合并首单和增长，强调自然成交 |

### 2.3 核心交互模式：对话式AI指导

每个阶段通过AI对话引导用户完成具体行动：
- AI给出当前阶段的行动指南和具体任务
- 用户回答问题、确认行动完成
- AI判断是否达到阶段标准，决定是否升级
- 用户始终知道：自己在哪一步、下一步做什么、怎么判断过了这步

### 2.4 阶段2-5的核心交付物

产品不替用户做内容，而是指导用户自己做：

| 阶段 | AI交付给用户什么 | 用户自己做什么 |
|------|-----------------|---------------|
| 2 冷启动 | 建账号checklist、首批内容选题建议、平台运营基础规则 | 自己建账号、自己拍/写内容、自己发 |
| 3 破千 | 数据分析建议、内容优化方向、选题策略 | 自己调整内容方向、自己持续发布 |
| 4 互动 | 回复话术模板、咨询转化流程、信任建立方法 | 自己回复评论/私信、自己引导加微信 |
| 5 成交 | 定价建议、交付清单、收款方式 | 自己谈价格、自己交付服务、自己收款 |

## 3. 功能需求

### 3.1 核心功能

#### FR-01: 对话式问答
- 用户通过聊天界面与AI交互
- 每次只问一个问题
- 支持预设选项和自由输入
- 记录所有回答用于后续分析

#### FR-02: 信心引擎
- 检测用户回答中的负面情绪
- 对负面情绪给出共情回复，引用用户之前提供的具体证据
- 对高质量回答给出正向反馈
- 反馈要求：引用用户原话，翻译成"值多少钱"或"能帮谁"

#### FR-04: 阶段管理
- 6阶段按顺序推进
- 每个阶段有明确升级标准
- 用户知道自己在哪一步、下一步做什么、怎么判断自己过了这步
- 阶段间数据传递：后续阶段读取前序阶段的输出

#### FR-05: AI内容生成
- 使用DeepSeek V4模型
- 生成内容必须接地气：使用目标用户行业的口语化表达
- 所有LLM输出为JSON格式
- LLM失败时降级：返回原始用户输入

#### FR-06: 进度展示
- 顶部进度条显示整体进度
- 阶段网格，显示当前阶段和升级标准
- 每个阶段内显示步骤进度

### 3.2 认证与用户管理

#### FR-07: 微信OAuth登录
- 用户通过微信授权登录
- 首次登录自动创建账户
- 获取微信头像和昵称

#### FR-08: 账户管理
- 查看账户信息
- 查看当前阶段和进度
- 删除账户（包括所有数据）

### 3.3 支付与付费墙

#### FR-09: 分层订阅（待定）
- 具体定价方案待验证后确定
- 免费层至少覆盖阶段0-1
- 付费层覆盖阶段2-5

#### FR-10: 微信支付集成
- 支持微信JSAPI支付
- 支付回调自动更新用户层级

### 3.4 非功能需求

#### NFR-01: 安全
- JWT认证保护所有API端点
- CORS限制为产品域名
- 用户敏感数据加密存储

#### NFR-02: 可用性
- 页面刷新不丢失进度
- LLM调用有重试机制
- 内容接地气，使用用户行业的口语化表达

#### NFR-03: 性能
- 单次LLM调用<10秒
- 页面加载<3秒
- 支持10个并发用户（MVP阶段）

## 4. 技术架构

### 4.1 技术栈
- **后端**: Python 3.14 + FastAPI
- **数据库**: SQLite (aiosqlite)
- **LLM**: DeepSeek V4 via EvoLink
- **认证**: 微信OAuth + JWT
- **支付**: 微信JSAPI支付
- **前端**: 原生HTML/CSS/JS（无框架）
- **部署**: Docker + 域名 firesing.cn

### 4.2 系统架构

```
微信浏览器
    ↓ 微信 OAuth
前端（H5）
    ├─ 对话界面（文字）
    └─ 阶段进度面板
    ↓ /api/chat
后端（FastAPI）
    ├─ SkillRunner（步骤引擎）
    │   ├─ SkillRegistry（6个阶段处理器）
    │   └─ StateManager（SQLite 状态持久化）
    ├─ PromptBuilder（模板系统）
    ├─ DeepSeekClient（LLM API）
    ├─ ConfidenceEngine（信心检测）
    └─ PaywallService（付费墙）
```

## 5. 验证计划

在写代码之前，先跑人工实验验证行为假设。

### 实验1：7天行动率
- 招15-20个目标用户
- 每天微信发1个极具体任务
- 判断标准：7天后连续完成5天以上的人低于20% → 纯软件日推不够

### 实验2：自动化边界
- A组：只给任务，不给内容
- B组：给任务 + 内容框架
- C组：给任务 + AI初稿，用户只需补行业细节
- 预测：C组最好

### 实验3：首笔收入验证
- 追踪14天内是否出现：私信咨询、加微信、首单
- 对照分析：拿到第一笔钱的人，后续留存是否显著更高

## 6. 成功标准

**Phase 1（人工陪跑验证，4周内）：**
- 实验1：20%以上用户连续5天完成任务
- 实验2：C组完成率显著高于A组
- 实验3：至少1个用户在14天内拿到首笔收入

**Phase 2（最小产品化，8周内）：**
- 6阶段成长路径完整上线
- 至少3个非刘为的用户通过系统完成阶段2（冷启动）
- 用户平均7天内完成首次发布

## 7. 失败标准

什么情况下承认这个方向不行：
- 实验1：7天后活跃率低于10%
- 实验2：A/B/C三组无显著差异
- 实验3：14天内零咨询/零成交
- 用户不愿按行动指南执行

## 8. 约束

- 用户技术能力有限
- AI生成的内容必须接地气
- 先人工陪跑验证，再产品化
- 不承诺收入结果
- 产品指导用户行动，不替用户做
