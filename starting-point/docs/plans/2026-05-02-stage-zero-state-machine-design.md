# Stage 0 State Machine Design

Generated by /brainstorming on 2026-05-02
Part of: 启点 v4.0 MVP

## Overview

Stage 0 is the discovery phase of the product. Through a 10-15 minute conversation, the AI identifies 3+ monetizable knowledge points from the user's industry experience.

This document specifies the technical design for Stage 0's conversation flow, completion detection, prompt strategy, error handling, and data model.

## Architecture

```
User → POST /api/chat {message}
         ↓
    ConversationEngine
         ↓
    Read conversation_state from SQLite
         ↓
    current_stage == 0 ?
    ├── No → route to correct stage handler
    └── Yes → StageZeroHandler
              ↓
         1. Load full message history from messages table
         2. Count user_messages
         3. If user_messages >= 10 → force-extract mode
         4. Build prompt (system + history + user message)
         5. Call LLM (single call)
         6. Parse response:
            ├── Plain text → save as assistant message, return to user
            └── JSON block → validate with Pydantic (StageZeroOutput)
                ├── Valid, >= 3 points → save extracted data, update stage=1, return summary
                ├── Valid, < 3 points → save extracted data, prompt LLM to continue
                └── Invalid JSON → retry up to 3 times, then fallback to text
         7. Save assistant message to messages table
         8. Save conversation_state to SQLite
         9. Return response
```

## Key Constants

| Constant | Value | Rationale |
|----------|-------|-----------|
| MAX_STAGE0_MESSAGES | 10 | Hard cap prevents runaway conversations. Most users complete in 5-8. |
| MIN_KNOWLEDGE_POINTS | 3 | Minimum for a meaningful product packaging in Stage 1 |
| MAX_LLM_RETRIES | 3 | Handles occasional malformed JSON without excessive latency |

## Prompt Design

### System Prompt (condensed structure)

```
你是一位经验丰富的中年人职业转型顾问。你正在和一个40+岁的人聊天，
ta有丰富的行业经验但不知道这些经验可以变现。

你的任务是通过对话识别ta的3-5个可变现知识点。

## 对话规则:
- 每次只问一个问题
- 根据用户的回答动态调整下一个问题
- 语气平等、尊重，像朋友聊天不是面试
- 如果用户回答模糊（"我什么都懂一点"），追问具体场景
- 最多问8-10个问题

## 何时输出结果:
当你确信已经识别出3个或更多可变现知识点时，用以下JSON格式输出:

[JSON schema embedded here — see Data Model section]

## 关键提醒:
- 只有在你确信有3+个有效知识点时才输出JSON
- 如果还不确定，继续提问
- 每个知识点必须是你从用户回答中直接推导出来的，不能臆造
```

### Force-Extract Prompt Override

When message count reaches the hard cap (10), the system appends:

```
对话已经进行了10轮。请现在基于已有的对话内容提取知识点。
如果不够3个，提取你找到的，并在summary中说明。
```

### Resume Prompt

When a user returns mid-conversation:

```
这是你之前和用户的对话。请继续从中识别可变现的知识点。
[Full message history inserted here]
```

## State Management

### Code-Side State (conversation_states.stage_data)

```json
{
  "schema_version": 1,
  "user_message_count": 5,
  "status": "collecting",
  "knowledge_points": [],
  "force_extract_triggered": false
}
```

Status values: `collecting` | `force_extract` | `completed`

- `collecting`: Normal conversation in progress
- `force_extract`: Hard cap reached, system forced extraction
- `completed`: Stage 0 done, knowledge points extracted, ready for Stage 1

### Conversation History

Full history stored in `messages` table. Sent to LLM every turn (no trimming needed — 10 messages at ~100 tokens each = ~1000 tokens, well within DeepSeek V4's 64K context window).

## Error Paths

### 1. LLM outputs JSON too early (< 3 valid points)

- Pydantic validation catches: `len(knowledge_points) < 3`
- System sends follow-up to LLM: "你提取的知识点不够3个，继续提问"
- LLM continues conversation

### 2. LLM never outputs JSON

- Hard cap at message 10 triggers force-extract mode
- System prompt changes to force extraction
- If still < 3 points: output partial results + honest feedback
- User gets: "我暂时还没找到你的经验里最适合变现的部分" + 2 options:
  - a) 再聊几轮换个角度
  - b) 先看看其他人是怎么做的（展示1-2个案例）

### 3. LLM outputs invalid JSON

- Pydantic validation fails
- Retry up to 3 times (resend same prompt)
- If all retries fail: discard JSON attempt, return last text response
- Log the failure for prompt debugging

### 4. User sends vague/generic answers

- LLM handles via prompt instruction ("追问具体场景")
- If after 5 rounds still vague: LLM naturally arrives at hard cap
- Falls into path 2 (force-extract with partial results)

### 5. Unexpected input (image, very long text, empty)

- Frontend validation: text input, max 500 chars, min 2 chars
- API validation: same rules server-side
- If somehow gets through: LLM handles gracefully (conversational model)

### 6. Mid-conversation resume

- Load conversation_state from SQLite
- Load all messages from messages table
- Reconstruct full history
- Resume from where user left off
- If partial knowledge_points exist, include in prompt context

## Data Model

### messages table

```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    user_id TEXT NOT NULL,
    role TEXT NOT NULL CHECK(role IN ('user', 'assistant', 'system')),
    content TEXT NOT NULL,
    stage INTEGER NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_messages_user_stage ON messages(user_id, stage);
```

### Pydantic validation models

```python
class KnowledgePoint(BaseModel):
    id: str = Field(pattern=r"^kp_\d+$")
    description: str = Field(min_length=10)
    industry: str = Field(min_length=1)
    knowledge_type: Literal[
        "price_transparency",
        "pitfall_guide",
        "channel_info",
        "industry_insider"
    ]
    target_buyer: str = Field(min_length=5)
    estimated_value: str = Field(min_length=5)

class StageZeroOutput(BaseModel):
    knowledge_points: list[KnowledgePoint] = Field(min_length=3)
    summary: str = Field(min_length=20)
```

## Design Decisions

1. **LLM-internal state (not code state machine)** — The "when to stop" decision is a judgment call that the LLM handles naturally. Code tracks message count for the hard cap but doesn't try to duplicate the LLM's understanding of conversation progress.

2. **Single LLM call per turn** — Cheaper and faster than two-call patterns. The prompt includes both conversation instructions and JSON output instructions, and the LLM chooses which to follow based on conversation state.

3. **Full history sent every turn** — With only 5-10 messages at ~100 tokens each, there's no need for trimming or summarization. This keeps the implementation simple.

4. **Pydantic validation + retry (not tool calling)** — Provider-agnostic, works with any LLM. Tool calling ties the product to DeepSeek's specific API.

5. **Force-extract at message 10** — Catches the case where the LLM never triggers JSON output. Most users complete in 5-8 messages, so 10 is a generous cap.
